# CGAlpha v3 — Cómo Funciona

## Arquitectura General

CGAlpha v3 es un **sistema de trading algorítmico con control de riesgos integrado** que opera en fases. La versión actual es **FASE 0** (Control Room con datos mock).

### Stack Tecnológico
```
┌─────────────────────────────────────┐
│  BROWSER (Frontend)                │
│  index.html │ app.js (Vanilla JS)  │
└─────────────────────────────────────┘
               │ HTTP/REST
               ▼
┌─────────────────────────────────────┐
│  SERVER.PY (Flask)                │
│  Auth Layer │ Routes │ Serialize   │
└─────────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│  DOMAIN → APPLICATION → SUBSYSTEMS │
│  Signal/Proposal → ExperimentRunner│
│  MemoryEntry → Library Manager     │
└─────────────────────────────────────┘
```

## Flujo Principal

1. **Usuario** interactúa con la **GUI** (navegador)
2. **GUI** llama a **API REST** con autenticación Bearer token
3. **Server** procesa requests y actualiza estado global
4. Cada acción genera **trazabilidad automática** en `memory/iterations/`

## Componentes Clave

### 1. **Control Room (GUI)**
- **Dashboard**: Estado del sistema, mercado mock, riesgos
- **Risk**: Parámetros de riesgo y Kill-Switch
- **Library**: Gestión de papers científicos
- **Theory Live**: Validación de hipótesis
- **Experiment Loop**: Backtesting walk-forward
- **Learning**: Memoria y promoción de estrategias

### 2. **Kill-Switch (2-pasos)**
```
POST /api/kill-switch/arm      → Solicita activación
POST /api/kill-switch/confirm → Confirma activación
POST /api/kill-switch/reset   → Rearma
```

### 3. **Library (Lila)**
Clasifica fuentes científicas:
- **Primary** (ev_level=1): Papers peer-reviewed
- **Secondary** (ev_level=2): Whitepapers, blogs técnicos
- **Tertiary** (ev_level=3): Foros, opiniones

### 4. **Experiment Loop**
Valida hipótesis con walk-forward (≥3 ventanas):
```python
Proposal → Backtest → Métricas → Promoción (si sharpe > 1.5)
```

### 5. **Memory Policy**
Jerarquía de conocimiento con TTL:
- **0a/0b**: RAW/NORMALIZED (automático)
- **1**: FACTS (Lila)
- **2**: RELATIONS (Lila)
- **3**: PLAYBOOKS (humano)
- **4**: STRATEGY (humano)

## Estado Global del Sistema

El sistema mantiene `_system_state` con:
- `phase`: "FASE_0" (actual)
- `status`: idle/running/degraded/error
- `kill_switch`: armed/triggered/disabled
- `circuit_breaker`: inactive/active
- `drawdown_session_pct`: Drawdown actual
- `data_quality`: valid/stale/corrupted

## Autenticación

Todos los endpoints requieren:
```bash
Authorization: Bearer <token>
```

Token default desarrollo: `cgalpha-v3-local-dev`

## Ejemplo de Uso

```bash
# 1. Iniciar servidor
python cgalpha_v3/gui/server.py

# 2. Verificar estado
curl -H "Authorization: Bearer cgalpha-v3-local-dev" \
     http://localhost:8080/api/status

# 3. Acceder GUI
# Navegador: http://localhost:8080
# Token: cgalpha-v3-local-dev
```

## FASE 0 vs Fases Futuras

**FASE 0 (actual):**
- Datos mock (simulados)
- Polling REST (cada 5s)
- Control Room operativa
- Sin trading real
- Sin WebSocket
- Sin multi-símbolo

**Fases futuras:**
- Conexión a exchanges reales
- WebSocket streaming
- Multi-símbolo
- Ejecución de alta frecuencia

La arquitectura está diseñada para evolucionar desde Control Room hacia sistema de trading completo manteniendo trazabilidad y controles de riesgo en cada fase.

# Integración de Signal Detector (Triple Coincidence) en CGAlpha v3

## Resumen del Signal Detector

El sistema detecta señales cuando **3 factores coinciden**:
1. **Vela Clave**: Alto volumen + cuerpo pequeño
2. **Zona de Acumulación**: Rango estrecho + volumen elevado
3. **Mini-Tendencia**: Pendiente clara + R² significativo

Score final = 70% componentes básicos + 30% factores avanzados

---

## Estrategia de Integración en v3

# ✅ El Oracle Sí Está en v3 (pero NO es Oracle Database)

El **Oracle** en CGAlpha v3 es un **componente de Machine Learning** que hace **Meta-Labeling**, no Oracle Database.

## Ubicación

## Qué es el Oracle

Es un **filtro probabilístico** que evalúa la calidad de una señal antes de ejecutarla:

```python
class OraclePrediction:
    trade_id: str
    confidence: float        # 0-1 (probabilidad de éxito)
    suggested_action: str    # "EXECUTE" | "IGNORE"
    estimated_delta_causal: float
```

## Función

**No predice dirección** (comprar/vender). **Predice calidad del contexto**:

```
Señal detectada → Oracle.predict() → ¿Es buena esta señal?
                           │
                           ├──► confidence > 0.70 → EXECUTE
                           └──► confidence < 0.70 → IGNORE
```

## Flujo con Oracle

```
SignalDetector.detect()
    │
    ▼
Señal generada
    │
    ▼
Oracle.predict(micro, signal_data)
    │
    ├──► Extrae features de microestructura (Trinity)
    ├──► Corre modelo.predict_proba()
    └──► Retorna OraclePrediction
         │
         ├──► confidence > 0.70 → Ejecutar trade
         └──► confidence < 0.70 → Ignorar señal
```
